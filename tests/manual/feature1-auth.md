# Feature 1 — User Registration and Login

**Endpoints under test:** `POST /auth/register`, `POST /auth/login`, `POST /auth/logout`

**Goal:** Verify that a user can create an account, sign in with valid credentials, and is correctly blocked when credentials or inputs are invalid.

| ID  | Test Case                                  | Condition Type        | Preconditions                                  | Steps                                                                                                          | Test Data                                                                                                  | Expected Result                                                                                                       |
| --- | ------------------------------------------ | --------------------- | ---------------------------------------------- | -------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| TC1 | Successful registration with valid data    | Valid input           | Email not yet registered                       | 1. Send POST `/auth/register` with valid payload<br>2. Observe response                                        | `{ "email":"alice@example.com", "password":"P@ssw0rd!", "firstname":"Alice", "lastname":"Wong" }`          | 201 Created. Response contains user id. New user record persisted. Success message displayed.                        |
| TC2 | Registration with already-used email       | State / data integrity | A user with `alice@example.com` already exists | 1. Send POST `/auth/register` reusing the existing email                                                        | Same payload as TC1                                                                                        | 409 Conflict (or 400). Clear error: "Email already registered". No duplicate row created.                            |
| TC3 | Registration with invalid email format     | Invalid input         | None                                           | 1. Send POST `/auth/register` with malformed email                                                              | `{ "email":"alice@", "password":"P@ssw0rd!", "firstname":"Alice", "lastname":"Wong" }`                     | 400 Bad Request. Validation error: "Invalid email format". User not created.                                          |
| TC4 | Registration missing required field        | Invalid input         | None                                           | 1. Send POST `/auth/register` omitting `firstname`                                                              | `{ "email":"bob@example.com", "password":"P@ssw0rd!", "lastname":"Smith" }`                                | 400 Bad Request. Validation error names the missing field. User not created.                                          |
| TC5 | Password at minimum boundary length        | Boundary values       | None                                           | 1. Register with the shortest allowed password<br>2. Then one character shorter (should fail)                   | Min: `Aa1!aaaa` (8 chars). Below min: `Aa1!aaa` (7 chars)                                                  | Min length → 201 Created. Below min → 400 Bad Request with length error.                                              |
| TC6 | Successful login with valid credentials    | Valid input           | TC1 user exists                                | 1. Send POST `/auth/login` with correct email and password                                                      | `{ "email":"alice@example.com", "password":"P@ssw0rd!" }`                                                  | 200 OK. Auth/session token returned. User is signed in and can access protected endpoints.                            |
| TC7 | Login with wrong password                  | Invalid input         | TC1 user exists                                | 1. Send POST `/auth/login` with wrong password                                                                  | `{ "email":"alice@example.com", "password":"WrongPass1!" }`                                                | 401 Unauthorized. Generic message: "Invalid email or password". No token issued.                                      |
| TC8 | Login with non-existent email              | Invalid input         | No user with `ghost@example.com`               | 1. Send POST `/auth/login` with unknown email                                                                   | `{ "email":"ghost@example.com", "password":"P@ssw0rd!" }`                                                  | 401 Unauthorized. Same generic message as TC7 (no user-existence leak). No token issued.                              |
| TC9 | Logout clears the session token            | Usability             | User signed in (TC6 token in hand)             | 1. Send POST `/auth/logout` with the token<br>2. Retry a protected endpoint with the same token                 | Authorization: Bearer `<token from TC6>`                                                                   | 200 OK on logout. Subsequent protected request returns 401. Client redirected to login page with confirmation toast. |
